# Testing this module with a few accounts

A fill-in checklist for `plan`/`apply`-testing the module against real Teleport +
AWS. Validation so far is `validate`-level only.

> **Tooling note:** this repo was validated with OpenTofu (`tofu` 1.12.2) and the
> `.terraform.lock.hcl` files were generated by it. If you run **Terraform**,
> re-run `terraform init` (re-resolves the locks) and `terraform validate`
> before trusting them. Commands below use `terraform`; swap in `tofu` if used.

## 1. Teleport cluster (one-time, shared by all accounts)

- [ ] **Proxy address** → `teleport_addr` in `terraform.tfvars`
      (e.g. `yourtenant.teleport.sh:443`).
- [ ] **Cluster is v18+** (provider pinned `>= 18.0`) and supports the `aws-ra`
      integration sub_kind.
- [ ] **Export the Roles Anywhere CA bundle** (no provider data source for this —
      mandatory out-of-band step; `plan` fails immediately without it):
      ```
      tctl auth export --type awsra > teleport-awsra-ca.pem
      ```
      Drop it in this `example/` dir (read via `file(...)` in main.tf).
- [ ] **Teleport identity file** for the provider (tbot/MachineID) →
      `teleport_identity_file` (default `identity`). The bot's role must be able
      to create/manage `integration` resources.

## 2. Per AWS account (repeat for each test account)

- [ ] **Account ID** (12 digits) — replaces the `111111111111` / `222222222222`
      placeholders.
- [ ] **An assumable admin role** in the account that can create IAM roles,
      Roles Anywhere trust anchors, and profiles. The example assumes
      `arn:aws:iam::<acct>:role/TerraformAdmin` — either create that role in each
      account or edit the `assume_role` block.
- [ ] **Region** — example hardcodes `us-east-1`; confirm Roles Anywhere is
      available/acceptable there.
- [ ] **Decide `target_roles`** — which IAM roles users assume in that account.

## 3. Base credentials & tooling

- [ ] AWS creds in your shell (env vars / named profile) allowed to
      `sts:AssumeRole` into each account's admin role.
- [ ] Network access to `terraform.releases.teleport.dev` (Teleport provider
      registry).

## 4. Wire up each account (two edits per account — see note below)

Terraform can't `for_each` over providers, so each account is two static blocks:

1. `providers.tf` — one aliased `provider "aws"` with the account's
   `assume_role { role_arn = ... }` and region.
2. `main.tf` — one `module "aws_<name>"` bound to that provider via
   `providers = { aws = aws.<alias> }`, plus an entry in the `accounts` map
   passed to `module "teleport_integrations"`.

## 5. Run

```
cd example
cp terraform.tfvars.example terraform.tfvars   # then fill it in
terraform init
terraform plan      # expect N aws-account modules + N integrations
terraform apply
```

## 6. Verify apps appear (~5 min after apply)

With `sync_enabled = true` (default), Teleport polls each account's RA profiles
~every 5 min and creates one app per matching profile:
```
tsh apps ls
tctl get apps
```
Grant access in a Teleport role via `app_labels`
(`teleport.dev/aws-roles-anywhere-profile-arn`) + `aws_role_arns` (use the
`target_role_arns` output), then `tsh apps login <app> --aws-role <name>`.

---

### Why there's no single "accounts list" (the (b) refactor)

On **stock Terraform** there's no way to drive accounts from one list/map:
provider blocks can't use `count`/`for_each`, and per the HashiCorp docs *"module
calls using `for_each` or `count` cannot pass different provider configurations to
different instances"* — the documented remedy is exactly the separate-`module`-
blocks pattern here. (OpenTofu 1.9+ does support provider `for_each`, but this is
a Terraform environment.)

What *does* reduce the boilerplate: move the per-account **data** (account ID,
region, target roles) into a single `locals { accounts = {...} }` block and have
each `module`/`provider` reference `local.accounts["prod"]`. The provider+module
skeleton still repeats per account, but the values live in one place. Ask if you
want this applied to the example.
